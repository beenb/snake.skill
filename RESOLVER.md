# 心愿.skill Resolver

> 借鉴 `gbrain` 的 Resolver 思路：**先路由，再判断优先级，再决定是否并行**。

---

## 路由总原则

当用户请求“蒸馏 / 提炼 / 用某种风格回答”时，不要直接开始蒸。

必须先完成三层判断：

1. **对象类型判断**：这是什么对象
2. **对象优先级判断**：本地 Skill / 本地知识 / 公开资料谁优先
3. **执行策略判断**：值不值得启用多 Agent

---

## 路由表

| 触发表达 | 对象类型 | 优先材料 | 输出重点 |
|---|---|---|---|
| `蒸馏 马斯克` / `蒸馏 哪吒` | 公开对象 | 本地已有 Skill / 公开资料 / 角色设定 / 访谈 / 语录 | 核心特质、风格、边界 |
| `蒸馏 产品经理思维` | 方法论 / 职业能力 | 本地 Skill / 公开方法论 / 用户补充案例 | 能力结构、应用场景、局限 |
| `蒸馏 我的导师` / `蒸馏 我同事` | 私人对象 | 本地 Skill / `knowledge/` / 用户补料 | 语言风格、认知方式、擅长与边界 |
| `蒸馏 某个 Skill` | Skill / 工作流 | 本地 Skill 文档、说明、示例 | 输入输出、关键步骤、可迁移模块 |
| `蒸馏 某个现场作品` | 比赛对象 | 本地 Skill / 作品描述 / 功能清单 / 演示文案 | 卖点、差异化、90 秒展示点 |
| `当前能力不够` / `需要别的 Skill` | 能力扩展判断 | 本地 Skill、MCP、外部 SkillHub | 补全能力、安装、沉淀 |
| `用 XX 的方式回答` | 快速蒸馏模式 | 已有蒸馏结论 / 本地 Skill / 最小材料 | 直接进入风格化回应 |

---

## 对象优先级判断

识别对象后，先按以下顺序检查：

1. **本地专门 Skill**
2. **本地知识 / 已有蒸馏结果**
3. **公开资料 / MCP / 联网搜索**
4. **外部 SkillHub 扩能**

核心规则：

- **本地 Skill 优先于公开搜索**
- **本地已有材料优先于重新联网收集**
- **只有本地不够时，才进入外部扩能**

---

## 公开对象路线

适用对象：

- 公众人物
- 虚构角色
- 神话角色
- 有明确公开资料的方法论

执行要求：

1. 先检查本地是否已有可直接处理该对象的 Skill
2. 若没有，再收集公开证据
3. 区分“高可信结论”和“弱推断”
4. 输出 `Compiled Truth + Evidence Capsule`

---

## 私人对象路线

适用对象：

- 用户身边的人
- 自创角色
- 小众对象
- 依赖用户上传材料的对象

执行要求：

1. 先检查本地是否已有专门 Skill
2. 若没有，优先读取 `knowledge/`
3. 若材料不足，分批提问
4. 不得假装自己知道不存在的事实

---

## Skill / 工作流路线

适用对象：

- 某个 Skill
- 某个工作流
- 某个项目方法
- 某种“能力结构”本身

执行要求：

1. 优先检查本地是否已有该 Skill 或相关 Skill
2. 不只看表面功能
3. 要拆出输入、输出、关键步骤、验证方式
4. 重点提炼“哪些能力可迁移到别处”

这一路特别适合比赛里的“把作品再蒸成新能力”。

---

## 现场比赛路线

适用对象：

- 在场选手作品
- 临场创意
- 赛题方案
- 90 秒展示对象

执行要求：

1. 优先检查本地是否已有相关 Skill 可直接接入
2. 重点蒸亮点，而不是面面俱到
3. 输出要服务现场表达
4. 明确：它最值得讲的 3 个点是什么

---

## 单 Agent / 多 Agent 判断

除了对象类型路由，还要额外判断：当前对象值不值得开多 Agent。

### 优先单 Agent

满足任意两条，优先单 Agent：

- 本地已有专门 Skill 可直接处理
- 信息来源单一且清晰
- 任务步骤少于 3 个非平凡步骤
- 并行不能明显提高准确度
- 并行不能明显减少总耗时或上下文负担

### 适合多 Agent

满足任意两条，可启用多 Agent：

- 需要同时处理公开资料与本地资料
- 需要把收集、蒸馏、验证拆成明显不同阶段
- 需要一边扩能力，一边收证据
- 单 Agent 会导致上下文过重或流程过长
- 多路材料需要合流后统一生成结论

推荐编排：

- `Agent-A`：公开资料 / MCP 搜索
- `Agent-B`：本地知识 / 本地 Skill / 用户补料
- `Agent-C`：Compiled Truth 汇总
- `Agent-D`：Verification 校准

其中：

- `Agent-A` 与 `Agent-B` 可并行
- `Agent-C` 在 A/B 合流后执行
- `Agent-D` 在 C 之后执行

如果并行收益不明显，**就不要启用多 Agent**。

---

## 能力扩展判断（横切层）

除了对象类型路由，还要额外做一次能力检查：

1. 当前 Skill 自己能否完成任务
2. 是否需要调用 MCP / 联网搜索
3. 是否已有本地 Skill 可接管这部分能力
4. 若没有，是否需要去外部 SkillHub（如 `skillhub.cn`）搜索候选 Skill

如果发现合适的新 Skill：

- 先做最小筛选
- 再安装
- 再登记为本地能力资产
- 下次类似任务优先复用

---

## 路由失败时怎么办

如果无法判断对象类型，不要硬猜。

先问：

```text
这个对象更偏“公开可查”，还是更依赖你提供的本地资料 / 个人描述？
```

拿到答案后再进入蒸馏。
